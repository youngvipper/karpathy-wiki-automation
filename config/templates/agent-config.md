# Agent 知识库自动化配置

> 通用配置模板，适用于任意 AI Agent。
> 请根据你的实际领域修改 `领域配置` 部分。

## 知识库路径

- **知识库根目录**：~/MyWiki

> 请修改为你的实际知识库路径。

## 领域配置

> 请根据你的实际需求定义领域。每个领域包含：名称、目录、关键词。

### 领域定义格式

```
领域名称：xxx
目录：raw/<domain>/、wiki/<domain>/
关键词：xxx、xxx、xxx
```

### 示例领域

#### 领域一：技术研究
- 目录：`raw/tech/`、`wiki/tech/`
- 关键词：代码、架构、设计模式、性能优化、框架、编程语言

#### 领域二：项目管理
- 目录：`raw/project/`、`wiki/project/`
- 关键词：需求、排期、复盘、迭代、团队、协作

#### 领域三：学习笔记
- 目录：`raw/learning/`、`wiki/learning/`
- 关键词：学习、总结、笔记、概念、读书、课程

> 请根据你的实际需求修改以上领域定义。

## 自动化规则

### 触发条件

当对话中出现以下内容时，**自动触发**知识库操作：

1. 包含领域关键词的内容
2. 包含高价值信号的内容（原创观点、实践经验、数据分析等）

### 知识筛选标准

#### 必须采集（高价值）
1. **原创观点**：用户自己的见解、分析、总结
2. **实践经验**：实际操作中发现的技巧、踩过的坑
3. **数据分析**：统计数据、实验结果、对比分析
4. **解决方案**：解决具体问题的方法和步骤
5. **最佳实践**：经过验证的有效方法

#### 可以采集（中价值）
1. **学习笔记**：学习过程中的总结和理解
2. **工具介绍**：新工具的使用方法和评价
3. **标准规范**：国标、行标、行业规范
4. **案例分析**：典型案例的分析和解读

#### 不采集（低价值/无价值）
1. **简单问答**：简单的"是/否"回答
2. **重复内容**：与已有知识重复的信息
3. **临时信息**：临时性的、无长期价值的内容
4. **情绪表达**：纯情绪化的表达，无实质内容
5. **碎片信息**：零散的、无法形成完整知识的信息

### 自动操作流程

#### 流程 1：对话中发现知识
```
用户发言
    ↓
Agent 自动识别
    ↓
判断是否高价值（筛选标准）
    ↓
[是] → 自动采集 + 编译 + 更新索引
[否] → 跳过，不采集
    ↓
回复用户确认
```

#### 流程 2：查询知识库
```
用户提问
    ↓
Agent 自动查询知识库
    ↓
引用相关文章
    ↓
生成答案
    ↓
保存查询结果（如果质量高）
```

#### 流程 3：深度研究
```
用户请求研究
    ↓
Agent 自动搜索
    ↓
筛选高质量结果
    ↓
自动采集 + 编译
    ↓
生成研究报告
```

## 使用示例

### 示例 1：自动采集高价值知识

**用户**：我最近在研究微服务架构，发现对于小团队来说，单体架构可能更合适。因为微服务带来的运维复杂度远超其收益。

**Agent 自动判断**：
- ✅ 包含原创观点（用户的研究发现）
- ✅ 包含实践经验（小团队的适用性）
- ✅ 高价值知识，应该采集

**Agent 自动执行**：
1. 创建 `raw/tech/2026-06-24_microservice-vs-monolith.md`
2. 编译成 `wiki/tech/microservice-vs-monolith.md`
3. 更新索引
4. 回复确认

### 示例 2：跳过低价值内容

**用户**：今天天气不错。

**Agent 自动判断**：
- ❌ 无知识价值
- ❌ 与任何领域无关
- ❌ 不应该采集

**Agent 回复**：
- 不采集，正常对话

### 示例 3：查询时自动引用

**用户**：微服务和单体架构的优缺点是什么？

**Agent 自动执行**：
1. 查询知识库
2. 引用 `[[tech/microservice-vs-monolith]]`
3. 生成答案
4. 保存查询结果（如果质量高）

## 注意事项

1. **不要过度采集**：只采集真正有价值的知识
2. **保持领域隔离**：不同领域不要混入
3. **定期维护**：每周检查一次知识库健康
4. **质量优先**：宁缺毋滥，质量比数量重要
5. **自定义领域**：根据你的实际需求修改领域配置
